Part 1

CSS Selectors

The most common way to locate elements

CSS Selectors: The Basics

CSS Selectors are the same selectors you use in CSS stylesheets. They're fast, readable, and widely supported.

Common CSS Selector Types:

  • #id - Select by ID
  • .class - Select by class name
  • tag - Select by HTML tag
  • [attribute] - Select by attribute
  • parent > child - Direct child combinator
  • ancestor descendant - Descendant combinator

ID Selectors - #id

Best Practice: IDs should be unique on a page, making them the most reliable selector!

HTML Example:

<button id="submit-btn">Submit</button> <input id="username" type="text">

Playwright Code:

# Using CSS selector with ID page.locator("#submit-btn").click() page.locator("#username").fill("john_doe") # Alternatively, use get_by_test_id for test IDs page.get_by_test_id("submit-button").click()

Class Selectors - .class

Caution: Classes can appear multiple times on a page. Be specific!

HTML Example:

<button class="btn btn-primary">Login</button> <button class="btn btn-secondary">Cancel</button>

Playwright Code:

# Single class page.locator(".btn-primary").click() # Multiple classes (element must have both) page.locator(".btn.btn-primary").click() # Be specific to avoid ambiguity page.locator("button.btn-primary").click()

Attribute Selectors - [attribute]

Select elements based on any attribute and its value.

Syntax Variations:

# Has attribute (any value) page.locator("[data-testid]") # Exact match page.locator("[type='submit']") # Contains value page.locator("[class*='btn']") # Starts with value page.locator("[href^='https']") # Ends with value page.locator("[src$='.png']")

Attribute Selectors: Real Example

HTML:

<button type="submit" data-action="login">Login</button> <input type="email" name="userEmail" placeholder="Email"> <a href="https://example.com/docs">Documentation</a>

Playwright Code:

# Select by type page.locator("[type='submit']").click() # Select by data attribute page.locator("[data-action='login']").click() # Select by name page.locator("[name='userEmail']").fill("user@example.com") # Select links to specific domain page.locator("[href^='https://example.com']").click()

Combining CSS Selectors

Combine selectors for more precise targeting

Common Combinations:

# Tag + class page.locator("button.btn-primary") # Tag + ID page.locator("input#username") # Tag + attribute page.locator("button[type='submit']") # Multiple attributes page.locator("input[type='text'][name='search']") # Class + attribute page.locator(".form-control[placeholder='Email']")

CSS Combinators

Navigate relationships between elements

Types of Combinators:

# Descendant (space) - any level below page.locator("div.container button") # Direct child (>) - immediate children only page.locator("form > button") # Adjacent sibling (+) - immediately after page.locator("label + input") # General sibling (~) - any sibling after page.locator("h2 ~ p")

Combinator Example

HTML Structure:

<div class="login-form"> <form> <label>Email</label> <input type="email" name="email"> <button type="submit">Login</button> </form> </div>

Playwright Code:

# Button inside login form page.locator(".login-form button").click() # Direct child button of form page.locator("form > button").click() # Input that follows label page.locator("label + input").fill("test@example.com")

Part 2

XPath Basics

Powerful but use sparingly

What is XPath?

XPath (XML Path Language) is a query language for selecting nodes in an XML/HTML document.

When to Use XPath:

  • ‚úÖ When CSS selectors can't express the relationship
  • ‚úÖ Navigating up the DOM tree (to parent)
  • ‚úÖ Complex text matching
  • ‚úÖ Selecting by position

Note: XPath can be slower than CSS and harder to read. Use it when necessary, but prefer CSS and Playwright's modern locators when possible.

XPath Syntax Basics

Absolute vs Relative Paths:

# Absolute path (starts from root) - AVOID! page.locator("xpath=/html/body/div/form/button") # Relative path (starts with //) - PREFERRED page.locator("xpath=//button[@type='submit']")

Never use absolute paths! They break with any DOM change.

Common XPath Patterns

# Select by tag page.locator("xpath=//button") # Select by attribute page.locator("xpath=//button[@type='submit']") # Select by ID page.locator("xpath=//*[@id='submit-btn']") # Select by class page.locator("xpath=//button[@class='btn-primary']") # Contains class (for multiple classes) page.locator("xpath=//button[contains(@class, 'btn-primary')]") # Select by text content page.locator("xpath=//button[text()='Login']")

XPath: Text Matching

XPath excels at finding elements by their text content

# Exact text match page.locator("xpath=//button[text()='Login']") # Contains text page.locator("xpath=//div[contains(text(), 'Welcome')]") # Starts with text page.locator("xpath=//h1[starts-with(text(), 'Chapter')]") # Case-insensitive (using translate) page.locator("xpath=//button[contains(translate(text(), 'LOGIN', 'login'), 'login')]")

Tip: Playwright's get_by_text() is usually better than XPath for text matching!

XPath Axes: Navigation

Navigate relationships in the DOM tree

# Parent page.locator("xpath=//button[@id='submit']/parent::form") # Ancestor page.locator("xpath=//button[@id='submit']/ancestor::div") # Following sibling page.locator("xpath=//label[text()='Email']/following-sibling::input") # Preceding sibling page.locator("xpath=//input[@name='email']/preceding-sibling::label") # Child page.locator("xpath=//form/child::button")

XPath: Real Example

HTML:

<div class="product-card"> <h3>Laptop</h3> <p class="price">$999</p> <button class="add-to-cart">Add to Cart</button> </div>

Playwright Code:

# Find button in the product card with specific title page.locator("xpath=//div[.//h3[text()='Laptop']]//button[@class='add-to-cart']").click() # Get price of product named Laptop price = page.locator("xpath=//div[.//h3[text()='Laptop']]//p[@class='price']").text_content() print(price) # $999

Part 3

Playwright Modern Locators

The recommended way to locate elements

Why Use Playwright's Modern Locators?

Modern locators are user-facing and accessible - they work the way users interact with your app!

Benefits:

  • ‚úÖ More resilient to DOM changes
  • ‚úÖ Better accessibility testing
  • ‚úÖ Self-documenting code
  • ‚úÖ Auto-waiting and auto-retrying
  • ‚úÖ Better error messages

Playwright Recommendation: Always prefer modern locators over CSS/XPath when possible!

get_by_role() - The Best Locator

Recommended: This is Playwright's #1 recommended locator strategy!

Common Roles:

# Buttons page.get_by_role("button", name="Login").click() # Links page.get_by_role("link", name="Contact Us").click() # Textboxes page.get_by_role("textbox", name="Email").fill("test@example.com") # Checkboxes page.get_by_role("checkbox", name="Accept terms").check() # Headings expect(page.get_by_role("heading", name="Welcome")).to_be_visible()

get_by_role() - Common ARIA Roles

# Navigation page.get_by_role("navigation") # Main content page.get_by_role("main") # Radio buttons page.get_by_role("radio", name="Option A").click() # Combobox (dropdown) page.get_by_role("combobox").select_option("value") # List items page.get_by_role("listitem").filter(has_text="Item 1") # Tables page.get_by_role("table") page.get_by_role("row") page.get_by_role("cell")

get_by_text() - Find by Visible Text

Locate elements by their text content

# Exact match (default) page.get_by_text("Login").click() # Substring match page.get_by_text("Welcome", exact=False).is_visible() # Case sensitive by default page.get_by_text("LOGIN") # Won't match "Login" # Find in specific container page.locator(".sidebar").get_by_text("Settings").click()

Use Case: Great for clicking links, finding headings, or verifying text content!

get_by_label() - Find Form Inputs by Label

Locate form inputs by their associated label text

HTML Example:

<label for="email">Email Address</label> <input id="email" type="email"> <label> Password <input type="password"> </label>

Playwright Code:

# Works with both explicit and implicit labels page.get_by_label("Email Address").fill("user@example.com") page.get_by_label("Password").fill("secret123") # Partial match page.get_by_label("Email", exact=False).fill("user@example.com")

get_by_placeholder() - Find by Placeholder

Locate inputs by their placeholder text

HTML Example:

<input type="text" placeholder="Search products..."> <input type="email" placeholder="Enter your email">

Playwright Code:

# Exact match page.get_by_placeholder("Search products...").fill("laptop") # Partial match page.get_by_placeholder("email", exact=False).fill("user@example.com")

Note: Prefer get_by_label() over placeholder when possible - it's more accessible!

get_by_alt_text() - Find Images

Locate images by their alt text

HTML Example:

<img src="logo.png" alt="Company Logo"> <img src="profile.jpg" alt="User profile picture">

Playwright Code:

# Click on an image (if it's clickable) page.get_by_alt_text("Company Logo").click() # Verify image is visible expect(page.get_by_alt_text("User profile picture")).to_be_visible() # Partial match page.get_by_alt_text("profile", exact=False)

get_by_title() - Find by Title Attribute

Locate elements by their title attribute (tooltip text)

HTML Example:

<button title="Close dialog">X</button> <a href="/help" title="Get help">?</a>

Playwright Code:

# Click by title page.get_by_title("Close dialog").click() page.get_by_title("Get help").click() # Partial match page.get_by_title("help", exact=False).click()

get_by_test_id() - Test-Specific IDs

Test IDs are attributes added specifically for testing. Use them when user-facing attributes aren't stable.

HTML Example:

<button data-testid="submit-button">Submit</button> <div data-testid="error-message">Invalid input</div>

Playwright Code:

# Using data-testid (default attribute) page.get_by_test_id("submit-button").click() # Verify error message expect(page.get_by_test_id("error-message")).to_contain_text("Invalid")

Chaining and Filtering Locators

Combine locators for precise targeting

# Chain modern locators page.get_by_role("navigation").get_by_role("link", name="Home").click() # Filter by text page.get_by_role("listitem").filter(has_text="Product A").click() # Filter by nested element page.get_by_role("listitem").filter( has=page.get_by_role("button", name="Buy") ).click() # Combine with CSS page.locator(".product-card").get_by_role("button", name="Add to Cart").click()

Part 4

Best Practices

How to choose the right locator

Locator Priority: What to Use First

Playwright's Recommended Priority (Best to Worst):

Priority Order:

  1. get_by_role() - ARIA roles & accessible names
  2. get_by_label() - Form fields with labels
  3. get_by_placeholder() - Form fields with placeholders
  4. get_by_text() - Text content
  5. get_by_test_id() - Test-specific data attributes
  6. CSS selectors - When above don't work
  7. XPath - Last resort for complex cases

Good vs Bad Locators

‚ùå BAD Locators

# Absolute XPath page.locator("xpath=/html/body/div[2]/form/button") # Generic classes page.locator(".btn.btn-1") # Position-based page.locator("button:nth-child(3)") # Implementation details page.locator(___PLACEHOLDER_0___submit()']")

‚úÖ GOOD Locators

# Role-based page.get_by_role("button", name="Submit") # Label-based page.get_by_label("Email") # Test ID page.get_by_test_id("submit-form") # Semantic CSS page.locator("#login-form button[type='submit']")

Avoid Brittle Selectors

Brittle selectors break easily when the UI changes!

What Makes a Selector Brittle?

  • ‚ùå Position-dependent (nth-child, nth-of-type)
  • ‚ùå Absolute paths
  • ‚ùå Auto-generated class names (e.g., css-abc123)
  • ‚ùå Too specific (long chains of classes)
  • ‚ùå Implementation details (framework-specific attributes)

Instead: Use semantic, user-facing attributes that survive refactoring!

Keep Selectors Simple

# ‚ùå TOO COMPLEX page.locator("div.container > div.row > div.col-md-6 > form > div:nth-child(2) > input") # ‚úÖ BETTER - Use unique attribute page.locator("[name='email']") # ‚úÖ BEST - Use label page.get_by_label("Email")

Rule of Thumb: If you can't explain your selector in one sentence, it's too complex!

When to Add Test IDs

Sometimes adding a test-specific attribute is the best option

Add Test IDs When:

  • ‚úÖ No stable user-facing attributes exist
  • ‚úÖ Text content changes frequently (translations, A/B tests)
  • ‚úÖ Multiple similar elements need distinction
  • ‚úÖ Complex dynamic components

Example:

# HTML # Playwright - stable across translations and redesigns page.get_by_test_id("checkout-submit").click()

Debugging Locators

Tools to help you find and test locators

Debugging Techniques:

# Count matching elements count = page.locator(".btn").count() print(f"Found {count} buttons") # Get all text contents buttons = page.locator("button").all_text_contents() print(buttons) # Highlight element page.locator("#submit").highlight() # Take screenshot with bounding box page.locator("#submit").screenshot(path="button.png")

Tip: Use Playwright Inspector (PWDEBUG=1) to test locators interactively!

Using Browser DevTools

Test selectors in the browser console

In Browser Console:

// Test CSS selector document.querySelector("#submit-btn") document.querySelectorAll(".btn") // Test XPath (more verbose) $x("//button[@type='submit']") $x("//button[text()='Login']") // Count matches document.querySelectorAll(".btn").length $x("//button").length

Shortcut: Right-click element ‚Üí Inspect ‚Üí Copy Selector/XPath (but don't blindly use it!)

Common Locator Pitfalls

‚ùå Pitfall #1: Relying on Order

page.locator("button").nth(2) breaks if buttons are reordered

‚ùå Pitfall #2: Generated Class Names

page.locator(".css-1dbjc4n-x9f619") changes on every build

‚ùå Pitfall #3: Overly Specific

page.locator("div > div > div > button.btn") too fragile

‚ùå Pitfall #4: Ignoring Accessibility

Missing the chance to improve both tests AND user experience

Real-World Example: Login Form

HTML:

<form id="login-form"> <label for="email">Email Address</label> <input id="email" type="email" placeholder="you@example.com"> <label for="password">Password</label> <input id="password" type="password"> <button type="submit">Sign In</button> <a href="/forgot">Forgot Password?</a> </form>

Best Approach:

page.get_by_label("Email Address").fill("user@example.com") page.get_by_label("Password").fill("secret123") page.get_by_role("button", name="Sign In").click() page.get_by_role("link", name="Forgot Password?").click()

Quick Reference Cheat Sheet

# Modern Locators (Preferred) page.get_by_role("button", name="Submit") page.get_by_label("Email") page.get_by_placeholder("Search...") page.get_by_text("Welcome") page.get_by_test_id("submit-btn") # CSS Selectors page.locator("#id") # ID page.locator(".class") # Class page.locator("[attr='value']") # Attribute page.locator("tag.class") # Combination # XPath (use sparingly) page.locator("xpath=//button[@type='submit']") page.locator("xpath=//button[text()='Login']") # Chaining page.locator(".container").get_by_role("button").click()

Summary

Key Takeaways:

  • ‚úÖ Prefer Playwright's modern locators (get_by_role, get_by_label)
  • ‚úÖ Use CSS selectors for stable attributes (ID, name)
  • ‚úÖ Use XPath only when necessary (parent navigation, complex text)
  • ‚úÖ Avoid brittle selectors (position, generated classes)
  • ‚úÖ Keep selectors simple and readable
  • ‚úÖ Add test IDs when user-facing attributes aren't stable
  • ‚úÖ Test your selectors in browser DevTools
