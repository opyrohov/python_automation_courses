What is a Class?

A class is a blueprint or template for creating objects.

Real-World Analogy:

  • üç™ Cookie cutter ‚Üí Class
  • üç™ Cookies ‚Üí Objects
  • üèóÔ∏è Blueprint ‚Üí Class
  • üè† Houses ‚Üí Objects

Code Example:

  • Dog class ‚Üí Blueprint
  • my_dog ‚Üí Specific dog
  • your_dog ‚Üí Different dog
  • Each has own data
üí° Key Point: Classes define structure, objects are actual instances!

Creating Your First Class

class Dog: """A simple dog class.""" pass # Empty class for now # Create instances (objects) my_dog = Dog() your_dog = Dog() print(my_dog) # <__main__.Dog object at 0x...>
Note: Use PascalCase for class names (capitalize each word)

The __init__ Method

__init__ is a special method called a constructor.

It runs automatically when you create a new object.

class Dog: def __init__(self, name, age): """Initialize a new dog.""" self.name = name self.age = age # Create dogs with different data buddy = Dog("Buddy", 3) max_dog = Dog("Max", 5) print(buddy.name) # Buddy print(max_dog.name) # Max

Understanding 'self'

self represents the current instance being created or used.

class Person: def __init__(self, name): self.name = name # THIS instance's name person1 = Person("Alice") # self = person1 person2 = Person("Bob") # self = person2
üí° Think of self as: "This specific object's..."
self.name = "This object's name"

Instance Variables

Instance variables are data that belongs to each specific object.

class Student: def __init__(self, name, grade): self.name = name # Instance variable self.grade = grade # Instance variable self.courses = [] # Instance variable student1 = Student("Alice", "A") student2 = Student("Bob", "B") # Each student has their own data print(student1.name) # Alice print(student2.name) # Bob

Instance Methods

Methods are functions that belong to a class.

class Dog: def __init__(self, name): self.name = name def bark(self): """Make the dog bark.""" print(f"{self.name} says Woof!") def sit(self): """Make the dog sit.""" print(f"{self.name} is sitting") dog = Dog("Buddy") dog.bark() # Buddy says Woof! dog.sit() # Buddy is sitting

Methods with Parameters

class BankAccount: def __init__(self, owner, balance=0): self.owner = owner self.balance = balance def deposit(self, amount): """Deposit money.""" self.balance += amount print(f"Deposited ${amount}") def get_balance(self): """Get current balance.""" return self.balance account = BankAccount("Alice", 1000) account.deposit(500) print(account.get_balance()) # 1500

Class vs Instance

Class

  • Blueprint/Template
  • Defined once
  • Creates objects
  • Example: Dog

Instance (Object)

  • Actual thing created
  • Can have many
  • Has own data
  • Example: my_dog
class Dog: # This is the class (blueprint) pass my_dog = Dog() # This is an instance your_dog = Dog() # This is another instance

Why Use Classes in Automation?

  • Organization: Group related functionality
  • Reusability: Write once, use many times
  • Maintainability: Easy to update and fix
  • Scalability: Build large test suites
  • Page Object Model: Industry standard
  • Clean Code: More readable tests

Working with Multiple Instances

class TestCase: def __init__(self, name, priority="medium"): self.name = name self.priority = priority self.status = "pending" def run(self): print(f"Running: {self.name}") self.status = "passed" # Create multiple test cases test1 = TestCase("Login Test", "high") test2 = TestCase("Logout Test", "medium") test3 = TestCase("Profile Test", "low") test1.run()

Methods Calling Other Methods

class ShoppingCart: def __init__(self): self.items = [] def add_item(self, item): self.items.append(item) self.update_count() # Call another method def update_count(self): print(f"Cart has {len(self.items)} items") def checkout(self): self.update_count() print("Processing checkout...") cart = ShoppingCart() cart.add_item("Laptop")

String Representation: __str__ and __repr__

class Product: def __init__(self, name, price): self.name = name self.price = price def __str__(self): """User-friendly string.""" return f"{self.name} - ${self.price}" def __repr__(self): """Developer-friendly representation.""" return f"Product('{self.name}', {self.price})" laptop = Product("Laptop", 999.99) print(laptop) # Laptop - $999.99

Data Validation in __init__

class User: def __init__(self, username, age): # Validate username if len(username) < 3: raise ValueError("Username too short") self.username = username # Validate age if age < 0 or age > 150: raise ValueError("Invalid age") self.age = age # This will raise an error # user = User("ab", 25) # ValueError: Username too short user = User("alice", 25) # Valid

Page Object Model (POM)

POM is a design pattern where each web page is a class.

Benefits:

  • ‚úÖ Selectors in one place
  • ‚úÖ Easy to maintain
  • ‚úÖ Reusable page actions
  • ‚úÖ Clean test code

Without POM:

page.fill("#username", "test") scattered everywhere!

Testing WITHOUT Page Objects

# Test 1 - Login page.goto("https://example.com/login") page.fill("#username", "testuser") page.fill("#password", "pass123") page.click("#submit") # Test 2 - Login again (duplicated code!) page.goto("https://example.com/login") page.fill("#username", "admin") page.fill("#password", "admin123") page.click("#submit") # Problem: If selector changes, update everywhere!
‚ö†Ô∏è Problems: Duplicated code, hard to maintain, selector changes affect many tests

Testing WITH Page Objects

class LoginPage: def __init__(self, page): self.page = page self.username_input = "#username" self.password_input = "#password" self.submit_button = "#submit" def login(self, username, password): self.page.fill(self.username_input, username) self.page.fill(self.password_input, password) self.page.click(self.submit_button) # Test 1 login_page = LoginPage(page) login_page.login("testuser", "pass123") # Test 2 login_page.login("admin", "admin123")

Step-by-Step: LoginPage Class

class LoginPage: """Page object for login page.""" def __init__(self, page): # Store the Playwright page self.page = page # Define all selectors in one place self.username_input = "#username" self.password_input = "#password" self.submit_button = "button[type='submit']" self.error_message = ".error"
üí° Tip: Store all selectors as instance variables for easy updates!

Adding Methods to LoginPage

class LoginPage: def __init__(self, page): self.page = page self.username_input = "#username" self.password_input = "#password" self.submit_button = "#submit" def enter_username(self, username): self.page.fill(self.username_input, username) def enter_password(self, password): self.page.fill(self.password_input, password) def click_submit(self): self.page.click(self.submit_button)

Complete login() Method

class LoginPage: def __init__(self, page): self.page = page self.username_input = "#username" self.password_input = "#password" self.submit_button = "#submit" def login(self, username, password): """Perform complete login action.""" self.page.fill(self.username_input, username) self.page.fill(self.password_input, password) self.page.click(self.submit_button) # Usage in tests login_page = LoginPage(page) login_page.login("testuser", "Pass123!")

BasePage Pattern (Advanced)

class BasePage: """Base class for all page objects.""" def __init__(self, page): self.page = page def navigate_to(self, url): self.page.goto(url) def get_title(self): return self.page.title() class LoginPage(BasePage): # Inherit from BasePage def __init__(self, page): super().__init__(page) # Call parent __init__ self.username_input = "#username"

Page Object Model Best Practices

  • One class per page: Each page gets its own class
  • Store selectors: All selectors in __init__
  • Descriptive methods: Use clear method names
  • Return values: Methods should return data when needed
  • No assertions: Keep assertions in tests, not page objects
  • Reusable actions: Create methods for common actions
üí° Remember: Page objects represent actions users can take!
