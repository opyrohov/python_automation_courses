## Слайд 3: Навіщо потрібен дебагінг?

Тести будуть падати. Питання в тому: **наскільки швидко ви зможете знайти баг?**

**Без навичок дебагінгу:**

- Дивитися на код годинами
- Додавати print() скрізь навмання
- Змінювати код методом проб і помилок

**З навичками дебагінгу:**

- Системний підхід
- Правильний інструмент для кожної ситуації
- Виправлення за лічені хвилини

Коли тест падає, найгірше, що можна зробити — це хаотично змінювати код, сподіваючись, що щось спрацює. Професійний підхід — це використання правильних інструментів для діагностики проблеми. Сьогодні ми розглянемо повний набір інструментів для дебагінгу Playwright-тестів — від візуального інспектора до трейсингу та логування.

## Слайд 5: Playwright Inspector

```bash
# Запуск з Playwright Inspector
# Windows CMD:
set PWDEBUG=1 && pytest test_file.py --headed

# Windows PowerShell:
$env:PWDEBUG="1"; pytest test_file.py --headed

# Linux/Mac:
PWDEBUG=1 pytest test_file.py --headed
```

Inspector відкривається автоматично. Ви можете:
- **Покроково виконувати** кожну дію Playwright
- **Вибирати локатори** натискаючи на елементи
- **Тестувати селектори** в консолі

Playwright Inspector — це головний інструмент візуального дебагінгу. Щоб його запустити, потрібно встановити змінну середовища `PWDEBUG=1` перед запуском тестів. На Windows CMD і PowerShell синтаксис відрізняється — зверніть на це увагу. Коли Inspector відкриється, ви побачите два вікна: браузер і панель Inspector. У панелі можна покроково виконувати дії, а також клікати на елементи сторінки, щоб автоматично згенерувати локатор для них.

## Слайд 6: page.pause() — точка зупинки

```python
def test_login(page: Page):
    page.goto("https://the-internet.herokuapp.com/login")
    page.locator("#username").fill("tomsmith")

    # ЗУПИНКА ТУТ - Inspector відкриється!
    page.pause()

    page.locator("#password").fill("SuperSecret!")
    page.locator("button[type='submit']").click()
```

⚠️ Працює тільки з `--headed`! Не працює в headless-режимі.

`page.pause()` — це як breakpoint у звичайному дебагері, тільки для Playwright. Ви вставляєте цю команду в будь-яке місце тесту, і коли виконання дійде до неї — тест зупиниться, відкриється Inspector, і ви зможете вручну досліджувати стан сторінки. Це надзвичайно зручно, коли потрібно зрозуміти, чому локатор не знаходить елемент або чому сторінка виглядає не так, як очікувалося. Обов'язково запускайте тести з прапорцем `--headed`, інакше pause не спрацює.

## Слайд 7: Умовна зупинка

```python
def test_with_conditional_pause(page: Page):
    page.goto(f"{BASE_URL}/login")
    page.locator("#username").fill("wrong_user")
    page.locator("#password").fill("wrong_pass")
    page.locator("button[type='submit']").click()

    # Зупинка тільки коли щось несподіване
    if "/secure" in page.url:
        print("Несподівано: логін пройшов!")
        page.pause()  # Дослідити стан
```

Зупиняйтесь тільки тоді, коли потрібно дослідити несподіваний стан.

Умовний `page.pause()` — це просунута техніка. Замість того, щоб зупинятися щоразу, ми додаємо умову. Наприклад, якщо ми тестуємо невалідний логін — система НЕ повинна пускати нас далі. Але якщо раптом ми потрапили на `/secure` — це означає, що щось пішло не так, і саме тоді ми зупиняємося для дослідження. Це дозволяє запускати тести автоматично, але зупинятися тільки при виявленні аномалій.

## Слайд 8: Уповільнене виконання

```python
# conftest.py
import pytest

@pytest.fixture(scope="session")
def browser_type_launch_args(browser_type_launch_args):
    return {
        **browser_type_launch_args,
        "slow_mo": 1000,  # 1 секунда між діями
    }
```

```bash
# Або через командний рядок:
pytest --headed --slowmo 1000
```

Slow motion дозволяє візуально відстежувати кожну дію крок за кроком.

Іноді тест виконується настільки швидко, що ви не встигаєте побачити, що відбувається. Параметр `slow_mo` додає затримку між кожною дією Playwright — наприклад, 1000 мілісекунд (1 секунда). Можна задати це через фікстуру `browser_type_launch_args` у `conftest.py` або просто через прапорець `--slowmo` в командному рядку. У поєднанні з `--headed` це дає можливість спостерігати за тестом у реальному часі.

## Слайд 10: Повідомлення консолі

```python
def test_console_logging(page: Page):
    # Слухати ВСІ повідомлення консолі
    page.on("console", lambda msg:
        print(f"  [{msg.type}] {msg.text}"))

    # Слухати JavaScript-помилки
    page.on("pageerror", lambda err:
        print(f"  [JS ERROR] {err}"))

    page.goto(f"{BASE_URL}/javascript_error")
```

Типи консолі: `log`, `warning`, `error`, `info`, `debug`

Playwright дозволяє перехоплювати повідомлення з консолі браузера прямо в Python-коді. Метод `page.on("console", ...)` реєструє callback-функцію, яка викликається при кожному повідомленні консолі. А `page.on("pageerror", ...)` ловить необроблені JavaScript-помилки. Це надзвичайно корисно для виявлення помилок на фронтенді, які можуть впливати на поведінку тестів. Наприклад, якщо JavaScript кидає помилку — кнопка може не працювати, і тест впаде.

## Слайд 11: Логування мережевих запитів

```python
def test_network_logging(page: Page):
    # Логувати всі запити
    page.on("request", lambda req:
        print(f"  → {req.method} {req.url[:60]}"))

    # Логувати невдалі відповіді (статус >= 400)
    page.on("response", lambda resp:
        print(f"  ← {resp.status} {resp.url[:60]}")
        if resp.status >= 400 else None)

    page.goto(f"{BASE_URL}/login")
```

Чудово підходить для виявлення 404-помилок, невдалих API-викликів або повільних запитів.

Мережеве логування — ще один потужний інструмент. Подія `"request"` спрацьовує при кожному HTTP-запиті, а `"response"` — при кожній відповіді. Ми можемо фільтрувати відповіді за статусом — наприклад, показувати тільки помилки (статус 400 і вище). Це допомагає швидко зрозуміти, чи проблема в тесті, чи на сервері. Наприклад, якщо API повертає 500 — це не баг тесту, а проблема бекенду.

## Слайд 12: Модуль logging в Python

```python
import logging

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s"

logger = logging.getLogger(__name__)

def test_with_logging(page: Page):
    logger.info("Starting login test")
    page.goto(f"{BASE_URL}/login")
```

Окрім `print()`, Python має повноцінний модуль `logging`. Його перевага — це рівні логування (DEBUG, INFO, WARNING, ERROR), автоматичні часові мітки та можливість запису у файл. Формат `%(asctime)s [%(levelname)s] %(message)s` дає зрозумілий вивід з часом і рівнем. Важливо: pytest за замовчуванням перехоплює вивід, тому для перегляду логів потрібно додати прапорець `-s`.

## Слайд 14: Що таке трейсинг?

Трейс записує **все** під час виконання тесту:

- Скріншоти на кожному кроці
- Знімки DOM
- Мережеві запити
- Повідомлення консолі
- Хронологія дій

Як «чорний ящик» для вашого тесту — переглядайте після падіння!

Трейсинг — це, мабуть, найпотужніший інструмент для посмертного аналізу тестів. Уявіть це як відеореєстратор у машині — він записує все, що відбувається, і ви можете переглянути запис після інциденту. Трейс зберігає скріншоти до і після кожної дії, повний стан DOM, усі мережеві запити та відповіді, повідомлення консолі. Потім цей файл можна відкрити в Trace Viewer і побачити покрокову хронологію тесту.

## Слайд 15: Ручний трейсинг у коді

```python
def test_with_trace(context: BrowserContext):
    # Почати запис
    context.tracing.start(
        screenshots=True,
        snapshots=True,
    )

    page = context.new_page()
    page.goto(f"{BASE_URL}/login")
    page.locator("#username").fill("tomsmith")

    # Зупинити і зберегти
    context.tracing.stop(path="trace.zip")
```

```bash
# Переглянути трейс:
playwright show-trace trace.zip
```

Для ручного трейсингу потрібен об'єкт `context` (BrowserContext). Метод `context.tracing.start()` починає запис, а `context.tracing.stop(path="trace.zip")` зупиняє і зберігає результат у ZIP-файл. Параметри `screenshots=True` і `snapshots=True` вмикають збереження скріншотів і знімків DOM відповідно. Після завершення тесту відкрийте файл командою `playwright show-trace trace.zip` — і ви побачите повну хронологію виконання.

## Слайд 16: Трейсинг через CLI

```bash
# Зберігати трейс тільки при падінні
pytest --tracing retain-on-failure --output results

# Зберігати скріншот тільки при падінні
pytest --screenshot only-on-failure --output results

# Записувати відео
pytest --video retain-on-failure --output results

# Всі артефакти при падінні
pytest --tracing retain-on-failure \
       --screenshot only-on-failure \
       --video retain-on-failure \
       --output results
```

`retain-on-failure` = зберігати тільки коли тест падає (економить місце на диску)

Це найзручніший спосіб використання трейсингу — через прапорці командного рядка. Не потрібно змінювати код тестів! Прапорець `--tracing retain-on-failure` автоматично записує трейс для кожного тесту, але зберігає файл тільки для тих, що впали. Аналогічно працюють `--screenshot` і `--video`. Параметр `--output` вказує директорію для збереження артефактів. Цей підхід ідеальний для CI/CD — артефакти падінь зберігаються автоматично.

## Слайд 19: TimeoutError

```python
# НЕПРАВИЛЬНО: селектор не знаходить нічого
page.locator("#wrong-id").click()  # TimeoutError!

# НЕПРАВИЛЬНО: елемент ще не завантажився
page.locator("#finish h4").text_content()  # TimeoutError!
```

**Причини:**
- Помилка в селекторі
- Елемент не завантажився
- Елемент всередині iframe

**Виправлення:**
- Перевірити в DevTools
- Додати `wait_for()`
- Використати `frame_locator()`

TimeoutError — це найпоширеніша помилка в Playwright-тестах. Вона виникає, коли Playwright не може знайти елемент за вказаним селектором протягом тайм-ауту (за замовчуванням 30 секунд). Найчастіша причина — банальна помилка в селекторі: неправильний id, пропущена крапка перед класом тощо. Друга причина — елемент ще не з'явився на сторінці. Третя — елемент знаходиться всередині iframe і до нього потрібно звертатися через `frame_locator()`.

## Слайд 20: Strict Mode Violation

```python
# НЕПРАВИЛЬНО: знайдено 2 чекбокси
page.locator("input").check()  # Error: strict mode violation!

# ВИПРАВЛЕННЯ 1: Використати .first
page.locator("input").first.check()

# ВИПРАВЛЕННЯ 2: Використати .nth()
page.locator("input").nth(0).check()

# ВИПРАВЛЕННЯ 3: Більш точний селектор
page.locator("input[type='checkbox']:first-child").check()
```

⚠️ Перевіряйте `locator.count()` щоб побачити скільки елементів знайдено!

Strict mode violation — друга за поширеністю помилка. Playwright працює в суворому режимі за замовчуванням: якщо локатор знаходить більше одного елементу, він відмовляється виконувати дію — бо не знає, який саме елемент ви маєте на увазі. Є три способи виправити це: `.first` бере перший елемент, `.nth(index)` бере елемент за індексом, або ви можете написати більш точний селектор. Перед виправленням завжди корисно перевірити `locator.count()` — скільки елементів знайдено.

## Слайд 21: iframe та відокремлені елементи

```python
# ПРОБЛЕМА: Елемент всередині iframe
page.locator("#tinymce")  # Не знайдено!

# ВИПРАВЛЕННЯ: Спочатку зайти у фрейм
frame = page.frame_locator("iframe#mce_0_ifr")
frame.locator("#tinymce").click()
```

```python
# ПРОБЛЕМА: Елемент видалено з DOM
btn = page.locator(".delete").first
btn.click()  # Видаляє сам себе
btn.click()  # Error: element detached!

# ВИПРАВЛЕННЯ: Повторний пошук після зміни DOM
remaining = page.locator(".delete")
assert remaining.count() == 0
```

Iframe — це окремий HTML-документ, вбудований у сторінку. Playwright не може знайти елементи всередині iframe через звичайний `page.locator()` — потрібно використовувати `page.frame_locator()`. Друга проблема — відокремлений елемент (detached). Якщо ви зберегли посилання на елемент, а потім він був видалений з DOM (наприклад, після кліку на кнопку «Видалити»), спроба взаємодії з ним викличе помилку. Рішення — зробити повторний пошук елемента.

## Слайд 22: assert проти expect()

```python
# НЕПРАВИЛЬНО: assert перевіряє миттєво — без очікування!
assert page.locator("#result").is_visible()  # Може бути False!

# ПРАВИЛЬНО: expect() автоматично очікує до тайм-ауту
expect(page.locator("#result")).to_be_visible(timeout=10000)
```

| | `assert` | `expect()` |
|---|---|---|
| Очікує? | Ні | Так (авто-повтор) |
| Для тайм-залежних? | Погано | Ідеально |
| Для значень? | Нормально | Нормально |

Це дуже важливе розмежування. Звичайний Python `assert` перевіряє умову миттєво — один раз. Якщо елемент ще не з'явився — `is_visible()` поверне `False`, і тест впаде. Натомість `expect()` від Playwright автоматично повторює перевірку протягом заданого тайм-ауту (за замовчуванням 5 секунд). Правило просте: для перевірки стану елементів на сторінці завжди використовуйте `expect()`, а звичайний `assert` — тільки для перевірки простих значень (наприклад, довжини списку).

## Слайд 24: Універсальна фікстура для дебагінгу

```python
@pytest.fixture
def debug_page(context: BrowserContext, request):
    context.tracing.start(screenshots=True, snapshots=True)
    page = context.new_page()

    page.on("console", lambda m: print(f"[{m.type}] {m.text}"))
    page.on("pageerror", lambda e: print(f"[JS ERROR] {e}"))

    yield page

    name = request.node.name
    if hasattr(request.node, "rep_call") and request.node.rep_call.failed:
        page.screenshot(path=f"debug/FAIL_{name}.png")
        context.tracing.stop(path=f"debug/FAIL_{name}.zip")
    else:
        context.tracing.stop()
```

Трейсинг + консоль + скріншот при падінні = одна фікстура

Ця фікстура — найпотужніший інструмент у вашому арсеналі. Вона об'єднує все в одному місці: автоматично запускає трейсинг, перехоплює повідомлення консолі та JS-помилки, а після завершення тесту перевіряє, чи тест впав. Якщо впав — зберігає скріншот і трейс з іменем тесту. Якщо пройшов — просто зупиняє трейсинг. Використовуйте `debug_page` замість `page` у тестах, де потрібна діагностика.

## Слайд 25: Хук для результату тесту

```python
# conftest.py - Потрібен для debug-фікстури
@pytest.hookimpl(tryfirst=True, hookwrapper=True)
def pytest_runtest_makereport(item, call):
    outcome = yield
    rep = outcome.get_result()
    setattr(item, f"rep_{rep.when}", rep)
```

Цей хук зберігає результат тесту на об'єкті item, щоб фікстура могла перевірити `request.node.rep_call.failed` після виконання тесту.

Фікстура `debug_page` потребує знати, чи впав тест. Але за замовчуванням pytest не передає цю інформацію фікстурам. Тому потрібен спеціальний хук `pytest_runtest_makereport`. Він перехоплює результат виконання тесту і зберігає його як атрибут на об'єкті `item`. Декоратор `@pytest.hookimpl(tryfirst=True, hookwrapper=True)` гарантує, що наш хук виконується першим і працює як wrapper навколо стандартного механізму.

## Слайд 27: Хелпер для дебагу елемента

```python
def debug_element(page: Page, selector: str):
    locator = page.locator(selector)
    count = locator.count()
    print(f"\n  Selector: '{selector}'")
    print(f"  Matches: {count}")
    if count > 0:
        print(f"  Visible: {locator.first.is_visible()}")
        print(f"  Enabled: {locator.first.is_enabled()}")
        print(f"  Text: '{locator.first.text_content()[:50]}'")
    else:
        print("  NOT FOUND!")
```

Використовуйте цю функцію, коли селектор працює не так, як очікувалося.

`debug_element()` — це ще один хелпер, який фокусується на конкретному елементі. Він показує, скільки елементів знайдено за селектором, чи видимий перший з них, чи він активний (enabled), і який у нього текст. Якщо нічого не знайдено — чітко повідомляє "NOT FOUND!". Це набагато інформативніше, ніж просто отримати TimeoutError без пояснення.

## Слайд 29: Чек-лист дебагінгу

| Перевірка | Інструмент / Виправлення |
|---|---|
| 1. Правильний селектор? | DevTools, `page.pause()`, `debug_element()` |
| 2. Проблема з таймінгом? | `wait_for()`, `expect()` |
| 3. Елемент в iframe? | `frame_locator()` |
| 4. Кілька збігів? | `.first`, `.nth()`, `.count()` |
| 5. Елемент відокремлений? | Повторний пошук локатора |
| 6. Проблема з мережею? | Логування request/response |

Цей чек-лист — ваш алгоритм дій при будь-якому падінні тесту. Проходьте пункти послідовно. Спершу перевірте селектор — це найчастіша причина. Потім — чи не проблема з очікуванням. Далі — iframe, множинні збіги, відокремлені елементи. І нарешті — мережеві проблеми. У 90% випадків причина знайдеться в перших трьох пунктах.
