Test Architecture

Building scalable test suites

Recommended Project Structure

my_project/ ‚îú‚îÄ‚îÄ tests/ ‚îÇ ‚îú‚îÄ‚îÄ conftest.py # Shared fixtures & hooks ‚îÇ ‚îú‚îÄ‚îÄ test_login.py # Login feature tests ‚îÇ ‚îú‚îÄ‚îÄ test_checkout.py # Checkout feature tests ‚îÇ ‚îî‚îÄ‚îÄ test_search.py # Search feature tests ‚îú‚îÄ‚îÄ pages/ ‚îÇ ‚îú‚îÄ‚îÄ base_page.py # Base page object ‚îÇ ‚îú‚îÄ‚îÄ login_page.py # Login page object ‚îÇ ‚îî‚îÄ‚îÄ checkout_page.py # Checkout page object ‚îú‚îÄ‚îÄ fixtures/ ‚îÇ ‚îî‚îÄ‚îÄ data_fixtures.py # Test data generators ‚îú‚îÄ‚îÄ test_data/ ‚îÇ ‚îî‚îÄ‚îÄ users.json # Static test data ‚îú‚îÄ‚îÄ pytest.ini ‚îî‚îÄ‚îÄ requirements.txt

Core Test Principles

1. One Test = One Thing

Each test verifies exactly one behavior

2. Independent

No test depends on another test's result

3. Arrange-Act-Assert

Setup ‚Üí Action ‚Üí Verify

4. Descriptive Names

test_login_with_valid_credentials_redirects

Arrange - Act - Assert

def test_successful_login(page: Page): # ARRANGE: Set up preconditions page.goto(f"{BASE_URL}/login") username = "tomsmith" password = "SuperSecretPassword!" # ACT: Perform the action page.locator("#username").fill(username) page.locator("#password").fill(password) page.locator("button[type='submit']").click() # ASSERT: Verify the outcome assert "/secure" in page.url

Clear structure makes tests easy to read and maintain.

Test Naming

Bad names:

def test_1(page): ... def test_login(page): ... def test_it_works(page): ...

Good names (what + expected):

def test_login_with_valid_credentials_redirects_to_secure(page): ... def test_login_with_invalid_username_shows_error(page): ... def test_checkbox_can_be_checked(page): ...

Part 2

Fixture Best Practices

Clean setup and teardown

Fixture Scope

Scope Created Use For
function Each test Page state, test data
class Each test class Shared class setup
module Each file Database connections
session Once Constants, config, base URL

Rule of thumb: use the narrowest scope possible. function is safest.

Yield Fixtures (Setup + Cleanup)

@pytest.fixture def authenticated_page(page: Page): # SETUP: Login page.goto(f"{BASE_URL}/login") page.locator("#username").fill("tomsmith") page.locator("#password").fill("SuperSecretPassword!") page.locator("button[type='submit']").click() yield page # ‚Üê Test runs here # CLEANUP: Logout (runs even if test fails!) logout = page.locator('a[href="/logout"]') if logout.is_visible(): logout.click()

Everything after yield = guaranteed cleanup.

Fixture Anti-patterns

Avoid:

  • Too many fixtures per test (max 3)
  • Session-scoped page fixtures (breaks isolation)
  • Side effects without cleanup
  • Importing conftest.py (it's auto-discovered)
  • Putting tests in conftest.py

Do:

  • Use yield for setup + cleanup
  • Compose fixtures (fixtures using fixtures)
  • Use factory fixtures for on-demand creation
  • Nest conftest.py for directory-specific fixtures

Part 3

Locator Best Practices

Selectors that don't break

Locator Priority

# Strategy Example Stability
1 get_by_role() get_by_role("button", name="Login") Best
2 get_by_text() get_by_text("Submit") Great
3 get_by_label() get_by_label("Username") Great
4 get_by_test_id() get_by_test_id("submit-btn") Good
5 locator("#id") locator("#username") Good
6 CSS / XPath locator("div.form input") Fragile

Selectors: Good vs Bad

Brittle (avoid):

# Position-dependent page.locator("div:nth-child(2) > form > input:first-child") # Deep XPath page.locator("//html/body/div[2]/div/form/input[1]")

Robust (prefer):

# Semantic page.get_by_role("button", name="Login") page.get_by_label("Username") # Stable attribute page.locator("#username")

Centralize Locators in Page Objects

class LoginPage: def __init__(self, page: Page): self.page = page # Locators defined ONCE self.username = page.locator("#username") self.password = page.locator("#password") self.submit = page.get_by_role("button", name="Login") def login(self, user: str, pwd: str): self.username.fill(user) self.password.fill(pwd) self.submit.click()

If a selector changes, update it in ONE place.

Part 4

CI/CD Integration

Automated test execution

GitHub Actions Workflow

# .github/workflows/tests.yml name: Playwright Tests on: [push, pull_request] jobs: test: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: python-version: "3.12" - run: pip install -r requirements.txt - run: playwright install --with-deps chromium - run: pytest --tracing retain-on-failure -v - uses: actions/upload-artifact@v4 if: failure() with: name: test-results path: test-results/

Docker Setup

# Dockerfile FROM mcr.microsoft.com/playwright/python:v1.50.0-noble WORKDIR /app COPY requirements.txt . RUN pip install -r requirements.txt COPY . . CMD ["pytest", "--tracing", "retain-on-failure", "-v"]
# Run tests in Docker docker build -t my-tests . docker run --rm -v ./results:/app/test-results my-tests

Parallel Execution

# Install pytest-xdist pip install pytest-xdist # Run with auto-detected workers pytest -n auto # Run with 4 workers pytest -n 4 # Combine with other flags pytest -n auto --tracing retain-on-failure -v

Tests MUST be independent for parallel execution!

CI Artifact Strategy

# Save everything on failure, nothing on success pytest --tracing retain-on-failure \ --screenshot only-on-failure \ --video retain-on-failure \ --output test-results/
Artifact Flag Size
Trace --tracing retain-on-failure ~2 MB
Screenshot --screenshot only-on-failure ~200 KB
Video --video retain-on-failure ~5 MB

Part 5

Reporting & Monitoring

Track and visualize results

Reporting Tools

# HTML Report pip install pytest-html pytest --html=reports/report.html --self-contained-html # JUnit XML (for CI dashboards) pytest --junitxml=reports/junit.xml # Allure Report (rich interactive) pip install allure-pytest pytest --alluredir=allure-results allure serve allure-results

Handling Flaky Tests

# Install retry plugin pip install pytest-rerunfailures # Retry failed tests 2 times pytest --reruns 2 --reruns-delay 5 # Mark specific flaky test @pytest.mark.flaky(reruns=3, reruns_delay=2) def test_sometimes_fails(page): ...

Better approach: Find and fix the root cause! Retries are a temporary solution.
